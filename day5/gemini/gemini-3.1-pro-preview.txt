### 1 и 4. Код решения с комментариями

```python
def twoSum(nums: list[int], target: int) -> list[int]:
    # Создаем словарь для хранения пройденных чисел и их индексов.
    # Ключ - само число, Значение - его индекс в массиве.
    num_map = {}
    
    # Проходим по массиву, получая одновременно индекс (i) и значение (num)
    for i, num in enumerate(nums):
        # Вычисляем разницу (число, которого не хватает до target)
        complement = target - num
        
        # Проверяем, встречали ли мы уже это недостающее число
        if complement in num_map:
            # Если да, возвращаем индекс найденного числа и текущий индекс
            return [num_map[complement], i]
        
        # Если нет, добавляем текущее число и его индекс в словарь
        # чтобы оно могло стать парой для следующих чисел
        num_map[num] = i
        
    # Возвращаем пустой список, если решение не найдено 
    # (хотя по условию гарантируется ровно одно решение)
    return []
```

---

### 2. Почему этот алгоритм имеет сложность O(n)

Классическое решение "в лоб" (brute force) использует два вложенных цикла, перебирая все возможные пары, что дает сложность $O(n^2)$. 

Наш алгоритм имеет временную сложность **$O(n)$**, потому что:
1. Мы проходим по массиву `nums` всего **один раз** с помощью цикла `for`.
2. Внутри цикла мы выполняем две основные операции:
   * Проверка наличия ключа в словаре `if complement in num_map`
   * Добавление элемента в словарь `num_map[num] = i`
3. В Python словари (хеш-таблицы) реализованы так, что поиск, вставка и удаление элементов происходят в среднем за константное время — **$O(1)$**.

Так как мы делаем $n$ итераций (где $n$ — длина массива), и на каждой итерации выполняем операции за $O(1)$, общая временная сложность составляет $n \times O(1) = O(n)$.

---

### 3. Анализ сложности

* **Временная сложность: $O(n)$**
  Как объяснено выше, мы делаем только один проход по массиву. Поиск разницы (`complement`) в хеш-таблице занимает в среднем $O(1)$ времени. В худшем случае (из-за коллизий хеширования) поиск может занять $O(n)$, но в Python хеш-функции для целых чисел оптимизированы, поэтому на практике это всегда $O(1)$.

* **Пространственная (память) сложность: $O(n)$**
  Для достижения скорости $O(n)$ мы жертвуем оперативной памятью. В худшем случае (когда подходящая пара состоит из первого и последнего элементов массива) нам придется сохранить в словарь `num_map` $n - 1$ элементов, прежде чем мы найдем решение. Таким образом, дополнительная память растет линейно пропорционально размеру входного массива.

### 5. Пример использования с тестовым случаем

```python
# Тестовый случай 1 (классический, исправленный на target = 9)
nums1 = [2, 7, 11, 15]
target1 = 9
print(f"Вход: nums = {nums1}, target = {target1}")
print(f"Выход: {twoSum(nums1, target1)}") 
# Вывод: [0, 1] (т.к. nums[0] + nums[1] = 2 + 7 = 9)

# Тестовый случай 2 (чтобы сумма была равна 19)
nums2 = [2, 7, 11, 15]

print(f"\nВход: nums = {nums2}, target = {target2}")
print(f"Выход: {twoSum(nums2, target2)}") 
# Вывод: [1, 2] (т.к. nums[1] + nums[2] = 7 + 11 = 18)

# Тестовый случай 3 (числа стоят не по порядку и есть отрицательные)
nums3 = [3, 2, 4]
target3 = 6
print(f"\nВход: nums = {nums3}, target = {target3}")
print(f"Выход: {twoSum(nums3, target3)}") 
# Вывод: [1, 2] (т.к. nums[1] + nums[2] = 2 + 4 = 6)
```
