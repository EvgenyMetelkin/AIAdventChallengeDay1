Вот оптимальное решение задачи **Two Sum** на Python с использованием хэш-таблицы (словаря). 

*Примечание к вашему примеру: в условии примера закралась опечатка (2 + 7 = 9, а не 19). В примере ниже я буду использовать `target = 9` для корректности математики, а также покажу пример для `target = 17` (2 + 15).*

### 1. Код решения с комментариями

```python
def two_sum(nums: list[int], target: int) -> list[int]:
    # Словарь для хранения пройденных чисел и их индексов.
    # Ключ — само число, значение — его индекс в массиве.
    num_to_index = {}

    # Проходим по массиву один раз, получая индекс (i) и само число (num)
    for i, num in enumerate(nums):
        # Вычисляем "дополнение" — число, которое нам нужно найти
        complement = target - num

        # Если дополнение уже есть в нашем словаре,
        # значит, мы нашли пару чисел, дающих в сумме target.
        if complement in num_to_index:
            # Возвращаем индекс найденного ранее числа и текущий индекс
            return [num_to_index[complement], i]

        # Если дополнения нет, сохраняем текущее число и его индекс в словарь
        num_to_index[num] = i

    # По условию задачи решение всегда существует, поэтому этот код не будет достигнут
    return []
```

---

### 2. Почему алгоритм имеет сложность $O(n)$?

Алгоритм работает за линейное время благодаря использованию структуры данных **Хэш-таблица** (в Python это встроенный тип `dict`).

1. **Однократный проход:** Мы итерируемся по массиву `nums` ровно один раз (один цикл `for`).
2. **Быстрый поиск за $O(1)$:** Вместо того чтобы искать дополнение (`complement`) во всем оставшемся массиве за $O(n)$ (как в брутфорс-решении), мы проверяем его наличие в словаре `num_to_index`. В среднем, поиск (`in`) и добавление нового элемента в хэш-таблицу в Python происходят за константное время — **$O(1)$**.
3. **Итог:** Мы делаем $n$ итераций, внутри каждой из которых выполняются операции за $O(1)$. Общая сложность составляет $n \times O(1) = O(n)$.

---

### 3. Анализ сложности

*   **Временная сложность (Time Complexity): $O(n)$**
    *   В худшем и среднем случае мы обходим массив из $n$ элементов один раз. Поиск каждого дополнения в словаре занимает $O(1)$.
*   **Пространственная сложность (Space Complexity): $O(n)$**
    *   В худшем случае (если искомая пара находится в самом конце массива) нам придется сохранить в словарь `num_to_index` почти все элементы массива. Для хранения $n$ элементов требуется $O(n)$ дополнительной памяти.
